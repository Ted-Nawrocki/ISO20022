Foundation

What Is ISO 20022?

ISO 20022 is an international standard for electronic data interchange between financial institutions. Published in 2004 by ISO Technical Committee TC68, it establishes a formal methodology for defining financial messages using a shared data dictionary, a business process model, and XML-based syntax. For corporate actions, it is the current messaging standard at DTC.

Think of ISO 20022 as a common language for financial messaging. Where the legacy ISO 15022 standard used proprietary, fixed-length field codes interpreted differently across institutions, ISO 20022 uses structured XML with fully spelled-out element names. A cash dividend under ISO 15022 carries the event type code DVCA. Under ISO 20022, the same event is identified as CashDividend—unambiguous to any system or operator reading the message.

The standard covers multiple financial domains: payments, securities settlement, asset servicing, and cash management. For corporate actions, the relevant domain is Securities Events, identified by the four-character code seev. Every corporate actions message issued by DTC begins with this prefix.

MT versus MX

ISO 20022 messages are commonly called "MX" messages to distinguish them from legacy "MT" (Message Type) messages defined under ISO 15022. MT564 and MT565 are ISO 15022 messages. CANO and CAIN are ISO 20022 MX messages. Both sets of shorthand appear throughout DTCC and SWIFT documentation; this guide uses both.

XML
Structured Data Format
ISO 20022 messages use Extensible Markup Language. Each element has a defined name, type, and validation rule, enabling precise machine-readable communication with no ambiguity in field interpretation.
SWIFT
Registration Authority
SWIFT serves as the ISO 20022 Registration Authority, maintaining the global message catalogue, validating change requests, and overseeing the annual Standard Release (SR) cycle.
STP
Straight-Through Processing
The richer, structured data of ISO 20022 enables higher automation rates, reducing the manual intervention and reconciliation burden that characterized processing under legacy formats.
SR
Annual Standard Release
Messages are updated through an annual Standard Release cycle. DTCC implements each SR. Version numbers in the MX Identifier may change with each release as field definitions and schemas are updated.
The Evolution · Part I

Before ISO 15022: The ISO 7775 Era

Understanding what came before ISO 20022 is the clearest way to appreciate why the migration happened.

ISO 7775, published in 1987, established the first internationally standardized framework for securities messaging on the SWIFT network. It defined message types and data structures for securities settlement, corporate actions, and related communications during the early period of electronic financial messaging.

ISO 7775 was a meaningful step forward, but it carried inherent limitations. Field definitions were inconsistently applied across markets. The standard lacked a rigorous underlying data model, which led to different institutions interpreting the same fields differently. As global securities markets grew in complexity and volume through the 1990s, these shortcomings became operationally significant—particularly in corporate actions, where event complexity and the volume of data required to describe an event were both increasing sharply.

The industry recognized the need for a more rigorous standard built on a formal data model with strict, consistent definitions. That recognition produced ISO 15022.

The Evolution · Part II

ISO 15022 and the MT Message Framework

ISO 15022, published in 1999, replaced ISO 7775 and introduced the MT (Message Type) framework that remains active in bilateral institutional messaging today. For corporate actions, it defined five core message types covering the full event lifecycle.

Message Type Name Purpose Typical Direction
MT564 Corporate Action Notification Announces a corporate action event: its type, terms, options, and key dates. The foundational announcement message. Custodian → Client
MT565 Corporate Action Instruction Conveys a participant's election or instruction for a voluntary or mandatory-with-choice event. Client → Custodian
MT566 Corporate Action Confirmation Confirms that a corporate action has been processed and a securities movement or cash payment has occurred or is pending. Custodian → Client
MT567 Corporate Action Status and Processing Advice Provides status information on a corporate action event or a previously received instruction. Serves both event-level and instruction-level status purposes. Custodian → Client
MT568 Corporate Action Narrative Provides free-form narrative related to a corporate action event when structured fields in the MT564 cannot carry all required information. Always used in conjunction with an MT564. Custodian → Client

ISO 15022's MT framework brought meaningful improvement over ISO 7775, but retained structural limitations. Individual messages were capped at 10,000 bytes, constraining the data any single message could carry. The data model relied heavily on coded qualifiers and abbreviations. Free-form narrative in the MT568 created reconciliation challenges. And as event complexity grew, the MT format increasingly required workarounds for data that did not fit cleanly within defined fields.

Still in use

ISO 15022 MT messages remain active in bilateral messaging between financial institutions. SWIFT has not established a universal retirement date for the standard in the corporate actions domain; European central securities depositories (CSD) members of the ECSDA have indicated they will not maintain the ISO 15022 format beyond 2030. However, DTC communicates exclusively in ISO 20022 MX messaging for all corporate actions data it distributes to and receives from participants.

The Evolution · Part III

The ISO 20022 Standard

ISO 20022 was published in 2004 by ISO Technical Committee TC68 as a methodologically rigorous replacement for the legacy MT framework. Its adoption across financial markets has been gradual, phased, and ongoing.

The core innovation of ISO 20022 is its methodology. Rather than defining messages first and constructing a data model afterward, ISO 20022 begins with a formal business process model and derives messages from it. Data elements carry precise, universally agreed definitions—not open to interpretation by individual institutions. The result is a standard where the same field means the same thing everywhere it appears.

ISO 20022 messages are referred to as "MX" messages and transmitted using XML syntax. The standard is organized into business domains; corporate actions fall within the seev domain, Securities Events. The seev message set covers the complete corporate actions lifecycle: announcements, instructions, status advisories, entitlement calculations, and confirmations.

SWIFT serves as the Registration Authority, receiving, validating, and publishing change requests to the message catalogue. The standard evolves through annual Standard Release cycles. DTCC participates in this process through the Securities Markets Practice Group (SMPG) and the International Securities Association for Institutional Trade Communication (ISITC), proposing and refining changes that reflect US market requirements. Changes are reviewed by the Securities Standards Evaluation Group (CA SEG)—an international body of industry experts—before incorporation into annual releases.

For the US market, DTC began its formal migration in the early 2010s under the Corporate Actions Transformation program. That program is covered in detail in the DTC Integration section of this guide.

The business area "seev" confirms this is a Securities Events message. The message identifier "031" maps to the CANO. The variant "001" identifies the core ISO standard; as of SR 2025, DTCC uses variant .001 across all core messages. Variant note: .002 to .001

Prior to SR 2025, DTCC used variant .002 across its core corporate actions messages. Variant .002 was the ISO 15022-compatible subset of ISO 20022, designed to ensure backward compatibility during the migration period. SR 2025 moved all DTCC core messages to variant .001. When reviewing older DTCC documentation or historical message logs, you will encounter the .002 variant identifier. The structural change from .002 to .001 represents DTCC's formal departure from the ISO 15022 compatibility requirement.

SEEV Messages · Announcement Group

Announcement Messages

Announcement messages initiate every corporate actions lifecycle. They carry event details from DTC outbound to participants. No announcement message flows inbound from a participant to DTC.

CANO
Corporate Action Notification
seev.031.001.15
The primary announcement message. DTC sends the CANO to all eligible participants when a corporate action event is confirmed. It contains the event type, terms, option details and option numbers, key dates (record date, ex date, payment date, instruction deadline), and an indicator of whether ISO 20022 instruction messaging is available for the event. The CANO is the foundational record for the event lifecycle.
DTC → Participant
CANO-E
Corporate Action Notification — Eligible Balance
DTC Extension
A DTC-specific follow-on to the CANO. Sent after DTC captures participant positions on the position capture date. The CANO-E provides the participant's specific eligible balance and available instruction options for that event. Unlike the CANO, which is broadcast to all eligible participants, the CANO-E is participant-specific. It is the signal that the instruction window is open.
DTC → Participant
CACN
Corporate Action Cancellation Advice
seev.039.001.xx
Notifies a participant that a previously announced corporate action event has been cancelled. Sent by DTC when an issuer withdraws or cancels an event prior to its completion. References the original CANO via the Corporate Actions Identifier (CAID). Any instructions previously submitted for the cancelled event are voided.
DTC → Participant
SEEV Messages · Instruction Group

Instruction Messages

Instruction messages are the only messages in the corporate actions lifecycle that flow inbound from participant to DTC. They are used exclusively for events requiring a participant election: voluntary events and mandatory-with-choice events.

CAIN
Corporate Action Instruction
seev.033.001.13
The primary instruction message. Sent by a participant to DTC to communicate an election for a corporate action event. A CAIN specifies the option number chosen (as defined in the CANO) and the quantity of eligible securities being tendered or elected. A CAIN may be submitted multiple times during the instruction period; the last valid submission before the cutoff governs.
Participant → DTC
CAIC
Corporate Action Instruction Cancellation Request
seev.040.001.xx
Sent by a participant to cancel a previously submitted CAIN prior to the instruction deadline. References the original CAIN by its transaction identifier. DTC responds with a CACS confirming or rejecting the cancellation. If the cancellation is accepted, the original instruction is voided and the participant must resubmit a new CAIN if they wish to maintain an election.
Participant → DTC
Instruction window

The instruction period is defined in the CANO by the instruction deadline (cutoff). Once the cutoff passes, DTC closes the instruction window and no further CAINs are accepted. For voluntary events, positions not covered by a valid CAIN before cutoff are not tendered and receive no processing. For mandatory-with-choice events, uncovered positions are processed under the default option identified in the CANO.

SEEV Messages · Status Group

Status Messages

Status messages flow from DTC to participants, confirming receipt and disposition of instructions, reporting event-level processing status, and providing statement-level reporting on instruction positions ahead of the deadline.

CAIS
Corporate Action Instruction Status Advice
seev.034.001.14
DTC's response to a received CAIN. Informs the participant whether the instruction was accepted, held as pending, or rejected, and provides the reason code for any rejection. A participant who receives a rejection must submit a corrected CAIN within the instruction period to have their election recorded.
DTC → Participant
CACS
Corporate Action Instruction Cancellation Request Status Advice
seev.041.001.xx
DTC's response to a CAIC. Confirms whether the cancellation request was accepted or rejected. If accepted, the original CAIN is voided and the position reverts to uninstructed. If rejected, the original instruction remains in force.
DTC → Participant
CAPS
Corporate Action Event Processing Status Advice
seev.032.001.08
Provides status information at the event level rather than the instruction level. Used by DTC to communicate lifecycle changes affecting all participants holding the security—such as event extensions, withdrawal of an event, or changes in event processing status. Distinguished from CAIS, which is instruction-specific.
DTC → Participant
CAST
Corporate Action Instruction Statement Report
DTC-specific
A DTC-specific reporting message sent in the days approaching the instruction deadline: from cutoff minus five through cutoff minus one. Provides a running statement of all instructions DTC has on record for that participant for a given event. Used for pre-cutoff reconciliation to ensure submitted positions match expectations before the window closes.
DTC → Participant
ACK / NAK

When a participant submits a CAIN, DTC performs an initial schema and subscription validation before issuing a CAIS. If the message fails technical format validation, DTC returns a NAK (Not Acknowledged). A NAK is a technical failure, not a business-level rejection—it means the message did not pass schema or subscription checks. An ACK confirms the message was received and queued for processing. The subsequent CAIS then communicates the business-level outcome: accepted, pending, or rejected.

SEEV Messages · Entitlement Group

Entitlement & Confirmation Messages

Entitlement and confirmation messages communicate the financial results of a corporate action. They cover projected allocations before the payable date, final confirmations after processing, and reversals when a confirmed movement must be unwound.

CAPA
Corporate Action Movement Preliminary Advice
seev.035.001.xx
A projected payment or allocation sent before the payable date. The CAPA communicates what a participant is expected to receive based on eligible positions, event terms, and (where applicable) submitted instructions. CAPA projections may be updated if event terms change prior to the payable date. Projection amounts of zero are valid—for example, when a rate has not yet been announced at the time of initial projection.
DTC → Participant
CAPC
Corporate Action Movement Preliminary Advice Cancellation Advice
seev.044.001.xx
Cancels a previously issued CAPA. Sent when DTC needs to retract a preliminary projection, typically due to a change in event terms or an event cancellation. References the original CAPA via the CAID. A new CAPA may follow if updated projection data becomes available.
DTC → Participant
CACO
Corporate Action Movement Confirmation Advice
seev.036.001.14
The final confirmation message. Sent by DTC after processing is complete to confirm the actual cash or securities movement to the participant's account. The CACO is the definitive record of entitlement and payment. It does not reproduce all CANO content; instead, it references the event via the CAID and COAF identifiers, keeping the message focused on movement data.
DTC → Participant
CARE
Corporate Action Movement Reversal Advice
seev.037.001.xx
Notifies a participant that a previously confirmed movement has been reversed. Sent by DTC in situations where a confirmed corporate action must be unwound—for example, due to a failed settlement, event error, or post-payment adjustment. References the original CACO and specifies the reason for the reversal.
DTC → Participant
SEEV Messages · Meetings

Meeting Messages

DTCC uses two meeting-specific messages for shareholder meeting notification and cancellation. These operate separately from the standard event lifecycle messages and are not part of the entitlement or instruction flows described above.

MTNO
Meeting Notification
seev.001 series
Notifies eligible participants of a shareholder meeting, including meeting type, date, agenda items, and voting options. Used for annual general meetings (AGMs), extraordinary general meetings (EGMs), and other shareholder votes. Provides the information participants need to submit proxies or voting instructions.
DTC → Participant
MTCN
Meeting Cancellation
seev.002 series
Notifies participants that a previously announced shareholder meeting has been cancelled or postponed. References the original MTNO. All voting instructions or proxy submissions previously submitted for the cancelled meeting are voided.
DTC → Participant

Event Playbooks · I

Voluntary Events

Voluntary events require affirmative action from a participant to participate. If no instruction is received prior to the deadline, the participant does not participate in the event and no processing occurs for that position.

Common voluntary events in the US market include tender offers, exchange offers, rights exercises, and warrant exercises. As a DTC participant sitting between your clients and DTC, participation in a voluntary event depends entirely on submitting a timely and valid CAIN before the DTC instruction deadline. Late instructions—submitted after cutoff—are not accepted regardless of circumstance.

The CANO announces the event and specifies option numbers. The CANO-E, distributed after DTC captures your eligible position, confirms your specific balance available for election. These two messages provide everything needed to construct and submit a CAIN.

Event Playbooks · II

Mandatory with Options Events (CHOS)

In DTCC's ISO 20022 implementation, this event category carries the processing code CHOS and the formal label Mandatory with Options. The industry shorthand "mandatory with choice" or "MWCO" refers to the same category and is widely used in operations contexts. Both terms appear throughout DTC documentation and industry practice.

CHOS events are mandated by the issuer but offer a choice of entitlement to the holder. The event will be processed for all eligible positions regardless of whether an instruction is received. A default option is always present: if no CAIN is submitted by the deadline, DTC processes the position under the default option identified in the CANO. Common examples include dividend with options, dividend reinvestment plans (DRIPs) where shareholders may elect cash instead of stock, and stock/cash elections in mergers. Every eligible holder receives a result. The instruction governs which option applies to a given position, not whether processing occurs at all.

The message flow for CHOS events is structurally identical to voluntary events. The operational distinction lies in the CANO itself, which identifies a default option number. DTC processes all uninstructed positions against that default after cutoff.

Event Playbooks · III

Mandatory Events

Mandatory events require no instruction from participants. DTC processes them automatically for all eligible positions. There is no instruction window, no CAIN, and no CAIS in the standard flow.

Common mandatory events in the US market include stock splits, reverse stock splits, cash dividends where no reinvestment option is offered, spin-offs, completed mergers where all shareholders receive uniform consideration, and name changes. The event occurs for every eligible holder. The participant's role is to receive the CANO, reconcile the CAPA projection against expected entitlements, and confirm the CACO against settlement records.

Event Playbooks · IV

Redemptions

Redemptions cover full and partial calls, scheduled maturities, and puts. They are primarily associated with fixed income securities—bonds, notes, and preferred instruments. Some redemptions are fully mandatory; others involve an instructable component where participants designate which positions to present.

For a full mandatory call or scheduled maturity, the message flow mirrors the mandatory event structure: CANO announces the redemption, CAPA projects the principal proceeds, and CACO confirms the payment. All eligible principal is redeemed without participant instruction.

For partial calls, the event may require participant instruction if the call is applied to specific positions or if DTC uses a lottery or pro-rata allocation methodology. In those cases, the lifecycle incorporates CANO-E, CAIN, and CAIS. The CANO will indicate whether the redemption event is eligible for ISO 20022 instruction messaging.

Event Playbooks · V

Income & Distributions

Income and distribution events include cash dividends, principal and interest (P&I) payments on fixed income securities, and other cash distributions. These are fully mandatory events. DTC calculates and distributes entitlements automatically based on eligible positions on the record date. No instruction is required or accepted.

Income events were the first category to go live under DTCC's CA Transformation program. Phase 3, completed in 2015, covered the full distribution lifecycle. The message flow is the most streamlined in the corporate actions suite.

The entitlement and allocation messages in this lifecycle—CAPA projections and CACO confirmations—are designed to be more compact than the CANO. They do not reproduce the full event details from the original announcement. Instead, they reference the event through the CAID and COAF identifiers, keeping each message focused on the specific movement data it is communicating.

Real-time vs. end-of-day batch

One of the concrete operational improvements ISO 20022 delivered for income processing at DTC is real-time messaging. Under the legacy proprietary DTC file formats (CCF files), participants received data in end-of-day batch files. ISO 20022 messaging is delivered in real time via IBM MQ, allowing participants to receive CANO announcements, CAPA projections, and CACO confirmations as they are generated—not at the end of a batch cycle.

DTC Integration

The CA Transformation Program

DTCC's Corporate Actions Transformation program was the multi-year initiative that brought ISO 20022 messaging to US market corporate actions processing. Understanding its history clarifies design decisions visible in the current system.

The program was driven by two realities. First, the global financial industry was converging on ISO 20022 as the common messaging language, and DTC's participants had expanded into global businesses requiring streamlined, internationally recognized communication standards. Second, the proprietary formats DTC had maintained since its founding in 1973 were not aligned with where the industry was heading. DTC worked with SMPG and ISITC throughout the program to propose changes to the ISO 20022 standard that reflected US market requirements, ensuring that DTC's implementation contributed to the global standard rather than simply adapting to it.

DTC Integration · CA Web

Corporate Actions Web (CA Web)

CA Web is DTC's browser-based platform for corporate actions processing. It provides a user interface for participants who need to view, manage, and respond to corporate actions. However, its scope has a significant and operationally critical boundary that every team member should understand clearly.

CA Web consolidated the distributions and redemptions lifecycle functionality that had previously been spread across legacy Participant Terminal System (PTS) and Participant Browser System (PBS) screens. For those event categories, participants may use CA Web directly, ISO 20022 MQ messaging, or a combination of both.

Voluntary reorganization instruction input is not available on CA Web. The DTCC Reorganizations Service Guide (updated March 2026) states this explicitly: "With the exception of voluntary reorganization instruction input, the Reorganization services concerning reorganization processing can be accessed on CA Web. Functionality for input of voluntary reorganization instructions can be accomplished via associated PTS and PBS functions, as well as via ISO 20022 messaging and APIs." Voluntary reorg instructions—including ATOP-eligible tender and exchange offers, rights subscriptions, and warrant exercises—must go through PTS PTOP, PBS Voluntary Tenders and Exchanges, or ISO 20022 Automated Instruction Messaging (CAIN via MQ).

For ATOP-eligible offers, the requirement is absolute. The Service Guide states that instructions submitted outside of PTS PTOP, PBS Voluntary Tenders and Exchanges, or Automated Instruction Messaging will not be accepted and will be rejected if submitted. CA Web can be used to view voluntary reorganization announcements and event details through its inquiry functions—but instruction submission for those events is not a CA Web capability.

Channel summary by event category

Distributions: CA Web, ISO 20022 MQ, or both. Redemptions: CA Web, ISO 20022 MQ, or both. Voluntary reorganizations (tender offers, exchange offers, rights subscriptions, warrant exercises): PTS PTOP, PBS Voluntary Tenders and Exchanges, or ISO 20022 Automated Instruction Messaging only. CA Web is used for announcement inquiry on reorg events, not for instruction input.

The DTCC Learning Center (dtcclearning.com) provides CA Web webinars, e-learning modules, and scenario-based guides for each event category. These resources are available to registered participants and complement the technical message specifications published on DTCC's corporate website.

DTC Integration · Extensions

US Market Extensions

ISO 20022 is a global standard, and not every data element required by the US market is defined within the core message schemas. DTCC addresses this through a "supplementary data" mechanism known as extensions.

Extensions are additional data structures appended to standard ISO 20022 messages to carry US market-specific information. They are not arbitrary additions. Extensions must be validated by the ISO 20022 community and formally registered in the ISO 20022 standard repository before DTCC may use them in production messages.

Examples of data carried in DTCC extensions include US market-specific tax indicators, processing codes, and event-type details for which no corresponding field exists in the global message definition. The Message Usage Guidelines and schema specifications available on DTCC's ISO 20022 Messaging Specifications page document the extension structure and field requirements for each message type.

Two key event identifiers

Every DTC corporate actions message references two identifiers that link lifecycle messages back to the original announcement. The CAID (Corporate Actions Identifier) is DTC's internal numeric identifier, assigned when the event is created in DTC's systems. The COAF (Official Corporate Action Event Identification) is the global cross-market identifier used for referencing the same event across different institutions. CAPA and CACO messages reference both identifiers rather than reproducing full CANO content, keeping downstream messages compact and focused. Understanding how to read and reconcile these identifiers is fundamental to tracking a corporate action across its full message lifecycle.

DTC Integration · Current Landscape

Current Landscape

As of the SR 2025 release, DTCC's corporate actions ISO 20022 messaging framework covers the full event lifecycle for distributions, redemptions, and reorganizations across all major event categories.

The SR 2025 release marked a structural milestone: all core messages were moved from variant .002 (the ISO 15022-compatible subset) to variant .001 (the main ISO 20022 standard). This represents DTCC's formal departure from the backward-compatibility requirement that had been maintained since the early phases of the CA Transformation. It signals industry confidence in ISO 20022 as the settled standard for US market corporate actions messaging.

DTCC continues to publish annual updates through the Standard Release cycle. Current SR message schemas, implementation documentation, and user guides are available on the DTCC website's ISO 20022 Messaging Specifications page. The DTCC Learning Center also provides regularly updated scenario-based guides covering each event category and message type.

For participants still using legacy proprietary DTC file formats such as CCF (Corporate Communications File) announcement files, DTCC has progressively reduced the availability of those formats. ISO 20022 MX messaging is the intended long-term standard for corporate actions communication at DTC.

Settlement modernization — a parallel stream

DTCC's modernization extends beyond corporate actions into settlement infrastructure. A separate Clearing and Settlement Transformation roadmap covers ISO 20022 adoption for settlement processing under a phased schedule running through 2026 and 2027. Current legacy transactional formats for settlement are targeted for decommission in Q3 2027. Corporate actions and settlement represent distinct implementation streams, but both are converging on ISO 20022 as the common standard across DTCC's participant-facing infrastructure.

Reference

Glossary

Key terms used throughout this guide and in DTCC and SWIFT corporate actions documentation.

ACK / NAK
Acknowledged / Not Acknowledged. DTC's initial response to any inbound message. A NAK signals a technical validation failure—schema or subscription error—not a business-level rejection. An ACK confirms the message was received for processing; the subsequent CAIS carries the business outcome.
CAID
Corporate Actions Identifier. DTC's internal numeric identifier assigned to each corporate action event at creation. Used across all lifecycle messages to link CAPA, CACO, and CARE back to the original CANO.
CA Transformation
DTCC's phased program to migrate all corporate actions processing from legacy proprietary formats to ISO 20022 MX messaging. Executed in five phases from approximately 2013 to 2019.
CA Web
Corporate Actions Web. DTC's browser-based platform for viewing and managing corporate action events. Launched during Phase 2 of the CA Transformation as a consolidated replacement for legacy PTS and PBS systems.
CAJWG
Corporate Actions Joint Working Group. The industry group within SMPG responsible for developing and maintaining global market practice principles for ISO 20022 corporate actions messaging.
CBPR+
Cross-Border Payments and Reporting Plus. SWIFT's usage guidelines governing ISO 20022 message usage for cross-border payments on the SWIFT network. Governs payments messaging, not securities event or corporate actions messaging.
CCF
Corporate Communications File. The legacy proprietary DTC batch file format used for announcing and reporting corporate action events prior to ISO 20022 adoption. End-of-day batch delivery; now being phased out in favor of real-time ISO 20022 MQ messaging.
COAF
Official Corporate Action Event Identification. The globally recognized event identifier used to cross-reference the same corporate action across different institutions. Complements the DTC-specific CAID in all lifecycle messages.
Extension
A supplementary data structure appended to a standard ISO 20022 message to carry data elements not defined in the core global schema. DTCC uses extensions for US market-specific data. Extensions must be formally registered with the ISO 20022 repository before use.
FIN / FINplus
SWIFT's legacy MT messaging service (FIN) and its MX-capable successor (FINplus). MT messages travel over FIN; MX messages travel over FINplus. DTCC uses IBM MQ rather than SWIFT FINplus for its participant-facing ISO 20022 messaging.
ISO 15022
International standard published in 1999 that defined the MT messaging framework for securities markets. Replaced ISO 7775. Defines MT564 through MT568 for corporate actions. Currently in coexistence with ISO 20022 in bilateral institutional messaging; no universal SWIFT retirement date has been set for the corporate actions domain.
ISO 20022
International standard published in 2004 by ISO Technical Committee TC68. Defines a modelling methodology and XML-based message format for financial communications across payments, securities, and other domains. The current standard for all DTC corporate actions messaging.
ISITC
International Securities Association for Institutional Trade Communication. An industry group that develops market practice standards for securities processing. Contributed US market input into the ISO 20022 corporate actions standard during the CA Transformation program.
Mandatory Event
A corporate action processed automatically for all eligible positions without requiring participant instruction. Examples: stock splits, cash dividends, completed mergers with uniform consideration. No CAIN is submitted; DTC generates CAPA and CACO based on eligible positions.
CHOS / Mandatory with Options
DTCC's ISO 20022 processing code and formal label for events that are mandated by the issuer but offer a choice of entitlement. The holder may submit a CAIN to elect a non-default option; without a CAIN, DTC processes the position against the default specified in the CANO. The industry shorthand "mandatory with choice" and the abbreviation MWCO refer to the same event category. Examples: dividend with options, DRIPs, stock/cash elections in mergers.
MQ Messaging
IBM MQ (Message Queue). The messaging middleware used by DTC for real-time delivery of ISO 20022 messages to and from participants. Replaced the end-of-day batch file model used under legacy proprietary CCF formats.
MT / MX
Message Type (MT) refers to ISO 15022 messages transmitted on the SWIFT FIN network (e.g., MT564). MX refers to ISO 20022 messages transmitted in XML format. MX is used informally to distinguish the new standard from legacy MT messages.
MX Identifier
The four-part identifier for an ISO 20022 message: BusinessArea.MessageID.Variant.Version (e.g., seev.031.001.15). Each segment carries specific meaning about the message's domain, function, variant, and release version.
seev
Securities Events. The ISO 20022 business area prefix for corporate actions messages. All DTCC corporate actions messages begin with "seev" in their MX Identifier.
SMPG
Securities Markets Practice Group. An international group of securities market practitioners that develops global market practice recommendations for ISO 20022 securities messaging, including corporate actions.
SR (Standard Release)
The annual maintenance release cycle for ISO 20022 messages, managed by SWIFT as Registration Authority. Each SR may introduce new message versions, field definition changes, or structural updates. DTCC implements each annual SR; version numbers in MX Identifiers may change with each release.
STP
Straight-Through Processing. Automated, end-to-end transaction processing without manual intervention. ISO 20022's structured data enables higher STP rates than legacy MT messaging by reducing field ambiguity and enabling more precise validation rules.
Variant
The third segment of an MX Identifier. Variant .001 is the main ISO 20022 standard message. Variant .002 was the ISO 15022-compatible subset used by DTCC prior to SR 2025 to maintain backward compatibility. As of SR 2025, all DTCC core messages use variant .001.
Voluntary Event
A corporate action requiring affirmative participant instruction to take effect. If no CAIN is submitted before the instruction deadline, no processing occurs for that position. Examples: tender offers, exchange offers, rights exercises, warrant exercises.
XML
Extensible Markup Language. The data format used by ISO 20022 MX messages. XML allows each data element to be explicitly named and validated, enabling structured, machine-readable messages with clear field boundaries—replacing the fixed-length, code-heavy field structure of ISO 15022 MT messages.
